Раздели крупный Next.js компонент на модули по методологии модульной архитектуры. Каждый модуль должен содержать примерно максимум 150 строк кода и иметь четкую зону ответственности.

Основные принципы разделения:

1. Разделение по ответственности - каждый модуль решает одну конкретную задачу
2. Инкапсуляция логики - бизнес-логика, UI логика и представление разделены
3. Однонаправленный поток данных - данные передаются сверху вниз через props
4. Переиспользуемость - модули должны быть независимыми и переиспользуемыми

Структура модулей:

- Главный компонент-контейнер - управляет состоянием и объединяет все модули
- UI компоненты - отвечают только за отображение, без логики
- Хуки - кастомные хуки для управления состоянием и side effects
- Утилиты - вспомогательные функции и хелперы
- Константы - строковые константы и настройки
- API функции - запросы к серверу и работа с данными

Требования к коду:

- Каждый модуль должен быть независимым и тестируемым
- Четкие интерфейсы между модулями
- Минимальная связность между модулями
- Максимальное сокрытие реализации
- Обработка состояний загрузки и ошибок
- Мемоизация где необходимо для оптимизации

Пример разделения функциональности:

- Управление состоянием выносится в кастомные хуки
- Логика форматирования данных - в утилиты
- API вызовы - в отдельные функции
- Валидация - в отдельные модули
- UI компоненты получают только готовые данные для отображения

Результат должен представлять из себя набор слабосвязанных модулей, которые легко поддерживать, тестировать и модифицировать независимо друг от друга.
Без использования TypeScript.
